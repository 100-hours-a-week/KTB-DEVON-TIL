# 부분 인덱스(Partial Index) 전략

> Devon의 조건부 인덱스 설계와 MySQL vs PostgreSQL 지원 차이 정리

## 들어가며

인덱스는 테이블의 모든 행을 대상으로 생성하는 것이 기본이다.
그런데 실제 쿼리가 항상 특정 조건을 만족하는 행만 조회한다면, 나머지 행에 대한 인덱스는 순수한 낭비다.

예를 들어 주문 테이블에서 `status = 'PENDING'`인 행은 전체의 5%인데,
이 쿼리를 위한 인덱스가 100%의 행을 모두 색인한다면 비효율적이다.

**부분 인덱스(Partial Index)** 는 특정 조건을 만족하는 행만 인덱스에 포함시키는 전략이다.

주요 이점:
- **인덱스 크기 감소**: 관련 없는 행을 제외해 인덱스가 작아짐
- **쓰기 성능 향상**: 인덱스 업데이트 대상 행이 줄어듦
- **캐시 효율 증가**: 작은 인덱스가 Buffer Pool에 더 잘 올라감

---

## 1. 개념 — 전체 인덱스 vs 부분 인덱스

### 전체 인덱스

```sql
-- 모든 행을 인덱스에 포함
CREATE INDEX idx_order_status ON orders (customer_id);
```

```
인덱스 내용:
customer_id=1, pk=10  (status=COMPLETED)
customer_id=1, pk=11  (status=COMPLETED)
customer_id=1, pk=12  (status=PENDING)    ← 실제로 조회할 행
customer_id=2, pk=20  (status=COMPLETED)
customer_id=2, pk=21  (status=COMPLETED)
... (전체 100% 저장)
```

### 부분 인덱스

```sql
-- PostgreSQL: WHERE 조건으로 인덱스 대상 행 제한
CREATE INDEX idx_order_pending ON orders (customer_id) WHERE status = 'PENDING';
```

```
인덱스 내용:
customer_id=1, pk=12  (status=PENDING만 포함)
customer_id=3, pk=31  (status=PENDING만 포함)
... (전체의 5%만 저장)
```

---

## 2. PostgreSQL — 네이티브 지원

PostgreSQL은 `CREATE INDEX ... WHERE` 구문으로 부분 인덱스를 직접 지원한다.

### 기본 사용법

```sql
-- 미처리 주문만 인덱싱 (전체의 5%라고 가정)
CREATE INDEX idx_pending_orders ON orders (customer_id)
WHERE status = 'PENDING';

-- 이 인덱스는 다음 쿼리에 사용된다
SELECT * FROM orders WHERE customer_id = 1 AND status = 'PENDING';
```

### Soft Delete 패턴

가장 흔한 사용 사례다.
삭제된 행은 대부분의 쿼리에서 제외되므로 인덱스에 포함할 필요가 없다.

```sql
-- 삭제되지 않은 행만 인덱싱
CREATE INDEX idx_active_posts ON posts (author_id)
WHERE deleted_at IS NULL;

-- 일반 조회에 사용
SELECT * FROM posts WHERE author_id = 1 AND deleted_at IS NULL;
```

전체 데이터의 10%가 soft delete 됐다면, 인덱스는 90% 크기로 유지된다.
시간이 지날수록 삭제 비율이 높아지면 인덱스 크기 절감 효과도 커진다.

### 부분 유니크 인덱스

"삭제되지 않은 행 중에서만 유일해야 한다"는 비즈니스 규칙을 인덱스로 표현할 수 있다.

```sql
-- 활성 상태인 사용자 이메일만 유니크 제약
CREATE UNIQUE INDEX idx_unique_active_email ON users (email)
WHERE deleted_at IS NULL;

-- 이메일이 같아도 한 명은 삭제, 한 명은 활성이면 허용됨
-- 두 명 모두 활성이면 에러
```

이 패턴은 애플리케이션 코드보다 인덱스에서 규칙을 강제하기 때문에 더 안전하다.

## 3. MySQL — 직접 지원 없음

MySQL(InnoDB)은 `WHERE` 조건이 있는 부분 인덱스를 지원하지 않는다.

```sql
-- MySQL에서 불가능
CREATE INDEX idx_pending ON orders (customer_id) WHERE status = 'PENDING';
-- ERROR: syntax error
```

### 대안 1: Generated Column + 인덱스

조건을 만족하지 않는 행에 대해 NULL을 반환하는 가상 컬럼을 만들고, 그 컬럼에 인덱스를 생성한다.
MySQL의 B-Tree 인덱스는 NULL도 저장하므로 **인덱스 크기는 줄지 않는다**.
하지만 **부분 유니크 제약** 구현에는 유용하다.

```sql
-- 삭제되지 않은 이메일만 유니크 제약 (MySQL 방식)
ALTER TABLE users
    ADD COLUMN active_email VARCHAR(255) AS (
        IF(deleted_at IS NULL, email, NULL)
    ) VIRTUAL;

CREATE UNIQUE INDEX idx_unique_active_email ON users (active_email);

-- active_email이 NULL이면 유니크 체크를 우회한다 (NULL != NULL)
-- 삭제된 사용자 이메일로 새 계정을 만들 수 있다
```

### 대안 2: 컴포지트 인덱스 순서 최적화

조건 컬럼을 인덱스 앞에 배치해 비효율적인 행에 대한 탐색을 줄인다.

```sql
-- 자주 필터링하는 status를 앞에 배치
CREATE INDEX idx_status_customer ON orders (status, customer_id);

-- MySQL 옵티마이저가 status='PENDING' 범위만 탐색한다
SELECT * FROM orders WHERE status = 'PENDING' AND customer_id = 1;
```

```
인덱스 구조:
CANCELLED, customer_id=1, pk=5   ← 탐색 제외
CANCELLED, customer_id=2, pk=8   ← 탐색 제외
PENDING,   customer_id=1, pk=12  ← 이 범위만 탐색
PENDING,   customer_id=3, pk=31
SHIPPED,   customer_id=2, pk=20  ← 탐색 제외
```

진짜 부분 인덱스처럼 물리적으로 분리되지는 않지만, B-Tree 범위 탐색으로 불필요한 행을 빠르게 건너뛴다.

### MySQL에서 부분 인덱스가 필요한 상황의 현실적 판단

| 필요한 것 | MySQL 접근 | 대안 |
|----------|-----------|------|
| 조건부 인덱스 크기 감소 | 불가 | 컴포지트 인덱스 순서 최적화 |
| 부분 유니크 제약 | Generated Column + NULL 트릭 | 애플리케이션 레벨 중복 검사 |
| Soft Delete 최적화 | `(deleted_at, 조회컬럼)` 컴포지트 | - |
| 복잡한 조건부 인덱스 | 불가 | PostgreSQL로 마이그레이션 고려 |

## 4. 실무 패턴

### Soft Delete 인덱스 설계

```sql
-- PostgreSQL (권장)
CREATE INDEX idx_posts_author_active ON posts (author_id, created_at)
WHERE deleted_at IS NULL;

-- MySQL (차선책)
-- deleted_at 컬럼을 복합 인덱스에 포함
CREATE INDEX idx_posts_author ON posts (deleted_at, author_id, created_at);

-- 쿼리
SELECT * FROM posts
WHERE deleted_at IS NULL AND author_id = 1
ORDER BY created_at DESC;
```

### 상태(Status) 컬럼 최적화

대부분의 행이 `COMPLETED` 상태이고, 실제 처리 대상은 `PENDING`인 케이스다.

```sql
-- PostgreSQL
CREATE INDEX idx_pending_jobs ON jobs (queue_name, created_at)
WHERE status = 'PENDING';

-- 이 쿼리에 최적
SELECT * FROM jobs
WHERE status = 'PENDING' AND queue_name = 'email'
ORDER BY created_at ASC
LIMIT 10;
```

전체 데이터의 5%만 인덱스에 들어가므로, 수천만 건 테이블에서도 인덱스가 수십만 건 수준으로 유지된다.

### JPA에서 부분 인덱스 선언

JPA 표준 `@Index`는 WHERE 조건을 지원하지 않는다.
DDL 초기화 방식 대신 Flyway/Liquibase로 직접 DDL을 작성하는 것이 현실적이다.

```java
// 이 방법으로는 부분 인덱스 생성 불가
@Table(indexes = @Index(columnList = "author_id"))  // WHERE 조건 없음
public class Post { ... }
```

```sql
-- V2__add_partial_index.sql (Flyway)
CREATE INDEX idx_posts_author_active ON posts (author_id, created_at)
WHERE deleted_at IS NULL;
```

## 핵심 원칙 정리

### 적용 조건
전체 행 중 특정 조건을 만족하는 비율이 낮을수록 부분 인덱스의 효과가 크다.
전체의 50% 이상이 조건을 만족한다면 일반 인덱스와 큰 차이가 없다.

### DB 선택
부분 인덱스가 핵심 최적화 수단이라면 PostgreSQL을 선택한다.
MySQL은 이 기능을 지원하지 않으며, 대안들은 완전한 대체가 되지 않는다.

### DDL 관리
JPA `@Index`로는 부분 인덱스를 선언할 수 없다.
Flyway 또는 Liquibase로 DDL을 직접 관리하는 것이 필수다.

## 마치며

부분 인덱스는 "이 쿼리가 항상 동일한 조건을 포함한다"는 사실을 인덱스 설계에 반영하는 것이다.
조건을 인덱스 밖에서 처리하는 게 아니라, 인덱스 자체가 그 조건을 이해하도록 만드는 방식이다.

Soft Delete 패턴을 쓰는 테이블에서 삭제된 데이터가 쌓일수록 인덱스 효율이 떨어지는 문제,
처리 완료된 작업이 대부분인 큐 테이블에서 미처리 작업 조회가 느려지는 문제,
이런 상황에서 부분 인덱스를 떠올릴 수 있어야 한다.

중요한 건 **쿼리 패턴과 데이터 분포를 함께 이해**하는 것이다.
