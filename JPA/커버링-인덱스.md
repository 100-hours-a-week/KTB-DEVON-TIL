# 커버링 인덱스(Covering Index) 전략

> Devon의 이중 조회 제거와 인덱스만으로 쿼리 완결하는 설계 정리

## 들어가며

인덱스를 사용한다고 해서 항상 빠른 것은 아니다.
인덱스로 조건에 맞는 행을 찾았더라도, 필요한 컬럼이 인덱스에 없으면 **실제 행 데이터를 다시 가져오는 추가 조회**가 발생한다.

이 두 번째 조회가 바로 성능 병목이 되는 지점이다.
특히 인덱스가 가리키는 행들이 디스크 여러 곳에 흩어져 있다면, 그만큼 랜덤 I/O가 발생한다.

**커버링 인덱스(Covering Index)** 는 쿼리에 필요한 모든 컬럼을 인덱스 자체에 포함시켜, 테이블 접근 없이 인덱스만으로 쿼리를 완결하는 전략이다.

---

## 1. InnoDB 세컨더리 인덱스의 이중 조회 구조

### 클러스터드 인덱스와 세컨더리 인덱스

InnoDB는 두 종류의 인덱스를 가진다.

**클러스터드 인덱스 (= 테이블 데이터 자체)**

```
PK=1 → { id:1, author_id:10, title:"foo", content:"...", created_at: 2024-01-01 }
PK=2 → { id:2, author_id:10, title:"bar", content:"...", created_at: 2024-01-02 }
PK=3 → { id:3, author_id:20, title:"baz", content:"...", created_at: 2024-01-03 }
```

**세컨더리 인덱스 (보조 인덱스)**

InnoDB의 모든 세컨더리 인덱스 리프 노드는 `[인덱스 컬럼값, PK값]`을 저장한다.

```
INDEX(author_id):
[author_id=10, pk=1]
[author_id=10, pk=2]
[author_id=20, pk=3]
```

### 이중 조회 발생 과정

```sql
SELECT title, created_at FROM post WHERE author_id = 10;
```

`INDEX(author_id)`만 있는 경우:

```
1단계 — 세컨더리 인덱스 스캔:
  author_id=10 → pk=1 발견
  author_id=10 → pk=2 발견

2단계 — 클러스터드 인덱스 조회 (테이블 접근):
  pk=1 → title="foo", created_at=2024-01-01 가져옴  (랜덤 I/O)
  pk=2 → title="bar", created_at=2024-01-02 가져옴  (랜덤 I/O)
```

조회한 PK가 많을수록, 그리고 해당 행들이 디스크에 흩어져 있을수록 2단계가 느려진다.
이 2단계 클러스터드 인덱스 조회를 **Bookmark Lookup** 또는 **이중 조회(Double Lookup)** 라고 한다.

## 2. 커버링 인덱스로 이중 조회 제거

`INDEX(author_id, title, created_at)`을 생성하면:

```
[author_id=10, title="foo", created_at=2024-01-01, pk=1]
[author_id=10, title="bar", created_at=2024-01-02, pk=2]
[author_id=20, title="baz", created_at=2024-01-03, pk=3]
```

인덱스 자체가 `title`과 `created_at`을 포함하기 때문에, 조회에 필요한 모든 데이터가 인덱스에 있다.

```
1단계 — 세컨더리 인덱스 스캔:
  author_id=10 → title="foo", created_at=2024-01-01 (여기서 끝!)
  author_id=10 → title="bar", created_at=2024-01-02 (여기서 끝!)

2단계 없음 — 테이블 접근 불필요
```

### EXPLAIN으로 확인

```sql
EXPLAIN SELECT title, created_at FROM post WHERE author_id = 10;
```

| id | type | key | Extra |
|----|------|-----|-------|
| 1 | ref | idx_author | **Using index** |

`Extra` 컬럼에 **`Using index`** 가 표시되면 커버링 인덱스가 동작하는 것이다.

커버링 인덱스가 아닐 때:
- `Using index condition` — 인덱스를 사용하지만 테이블 접근 필요
- (아무것도 없음) — 인덱스 없이 풀 테이블 스캔

## 3. 커버링 인덱스 설계 방법

### 기본 원칙: (조건 컬럼 → 정렬 컬럼 → 조회 컬럼)

```sql
SELECT title, created_at          -- 3. 조회 컬럼
FROM post
WHERE author_id = 10              -- 1. 조건 컬럼 (= 조건, 가장 앞)
ORDER BY created_at DESC;         -- 2. 정렬 컬럼
```

```sql
CREATE INDEX idx_covering ON post (author_id, created_at, title);
--                                  조건 컬럼   정렬 컬럼   조회 컬럼
```

### InnoDB는 PK를 항상 포함한다

InnoDB 세컨더리 인덱스는 PK를 자동으로 포함하기 때문에, `SELECT id`는 별도 선언 없이 항상 커버링 인덱스가 된다.

```sql
-- INDEX(author_id)만 있어도 커버링 인덱스
SELECT id FROM post WHERE author_id = 10;
-- 인덱스에 이미 [author_id, pk(id)]가 있다
```

### 예시: 목록 조회 최적화

```sql
-- 페이지네이션 목록 쿼리
SELECT id, title, created_at
FROM post
WHERE author_id = 10
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 20;
```

```sql
-- 커버링 인덱스
CREATE INDEX idx_post_list ON post (author_id, deleted_at, created_at, title);
```

| 목적 | 컬럼 | 이유 |
|------|------|------|
| = 조건 | `author_id` | 카디널리티 높은 조건 컬럼 앞에 |
| = 조건 | `deleted_at` | NULL 필터링 |
| 정렬/범위 | `created_at` | ORDER BY 처리 |
| SELECT | `title` | 조회 컬럼, 맨 뒤 |
| PK | `id` | InnoDB가 자동 포함 |

## 4. 커버링 인덱스가 특히 효과적인 상황

### 페이지네이션 — Deferred Join 패턴

OFFSET이 큰 페이지네이션은 뒤로 갈수록 느려진다.
커버링 인덱스와 Deferred Join을 조합하면 이 문제를 크게 개선할 수 있다.

```sql
-- 나쁜 예: OFFSET 100000이면 100000개 행을 모두 fetch한 뒤 버림
SELECT id, title, content, created_at
FROM post
WHERE author_id = 10
ORDER BY created_at DESC
LIMIT 20 OFFSET 100000;
```

```sql
-- 좋은 예: 커버링 인덱스로 PK만 먼저 추리고, 필요한 행만 조인
SELECT p.id, p.title, p.content, p.created_at
FROM post p
INNER JOIN (
    SELECT id
    FROM post
    WHERE author_id = 10
    ORDER BY created_at DESC
    LIMIT 20 OFFSET 100000
    -- ↑ 이 서브쿼리는 INDEX(author_id, created_at)로 커버링 인덱스 동작
    --   id(PK)만 반환하므로 테이블 접근 없음
) AS t ON p.id = t.id;
```

서브쿼리는 커버링 인덱스로 PK 목록만 빠르게 추출하고,
실제 행 데이터는 20개 PK에 대해서만 가져온다.

### 집계 쿼리

```sql
-- INDEX(author_id, status, created_at) 커버링 인덱스
SELECT COUNT(*), MIN(created_at), MAX(created_at)
FROM post
WHERE author_id = 10 AND status = 'PUBLISHED';
-- 테이블 접근 없이 인덱스만으로 집계 가능
```

## 5. 주의사항과 트레이드오프

### 인덱스 크기 증가

커버링 인덱스는 조회 컬럼까지 인덱스에 포함하므로 인덱스가 커진다.
`content TEXT` 같은 대용량 컬럼은 커버링 인덱스에 포함하지 않는 것이 좋다.

```sql
-- 나쁜 예: content가 크면 인덱스도 비대해짐
CREATE INDEX idx_bad ON post (author_id, title, content);

-- 좋은 예: content는 제외, 어차피 SELECT에서 테이블 접근이 필요하면 다른 방법 고려
CREATE INDEX idx_good ON post (author_id, title);
```

### 쓰기 성능 저하

인덱스에 포함된 컬럼이 UPDATE될 때마다 인덱스도 함께 갱신된다.
컬럼이 많을수록 쓰기 비용이 증가한다.

```
일반 인덱스 INDEX(author_id):
  title 변경 시 → 이 인덱스는 갱신 불필요

커버링 인덱스 INDEX(author_id, title):
  title 변경 시 → 인덱스도 함께 갱신 필요
```

쓰기 빈도가 높은 컬럼을 커버링 인덱스에 포함할 때는 쓰기 비용 증가를 고려한다.

### 인덱스 수 관리

커버링 인덱스를 쿼리마다 만들면 인덱스가 폭발적으로 늘어난다.
인덱스도 데이터 파일이므로 INSERT/UPDATE/DELETE 시 모두 갱신 비용이 발생한다.

하나의 커버링 인덱스로 여러 쿼리를 커버할 수 있도록 설계한다.

```sql
-- 이 인덱스 하나로 아래 두 쿼리 모두 커버링 가능
CREATE INDEX idx_post ON post (author_id, created_at, title);

-- 쿼리 1: title만 필요
SELECT title FROM post WHERE author_id = 10 ORDER BY created_at DESC;

-- 쿼리 2: created_at 범위 조건
SELECT title, created_at FROM post
WHERE author_id = 10 AND created_at > '2024-01-01';
```

## 핵심 원칙 정리

### 이중 조회가 발생하는 조건
세컨더리 인덱스로 행을 찾은 후, SELECT에 필요한 컬럼이 인덱스에 없으면 PK를 통해 클러스터드 인덱스를 다시 조회한다. 이것이 이중 조회다.

### 커버링 인덱스 설계 순서
`(= 조건 컬럼, 범위/정렬 컬럼, 나머지 SELECT 컬럼)` 순서로 구성한다. PK는 InnoDB가 자동 포함하므로 생략해도 된다.

### EXPLAIN 확인
`Extra: Using index` 가 표시되면 커버링 인덱스가 작동하는 것이다. 대용량 테이블의 조회 최적화 시 반드시 EXPLAIN으로 검증한다.

### Deferred Join
페이지네이션에서 OFFSET이 클 때, 커버링 인덱스 서브쿼리로 PK만 먼저 추리고 실제 데이터를 그 PK로 조인하는 패턴이 효과적이다.

## 마치며

커버링 인덱스는 인덱스를 "조건 탐색 도구"가 아닌 "데이터 저장소"로 활용하는 관점의 전환이다.
인덱스가 쿼리에 필요한 모든 것을 가지고 있다면, 테이블 접근 자체를 없앨 수 있다.

EXPLAIN을 습관적으로 확인하다 보면 `Using index condition`과 `Using index`의 차이가 눈에 들어오기 시작한다.
그 차이를 만드는 것이 커버링 인덱스 설계의 핵심이다.

중요한 건 **읽기 최적화와 쓰기 비용의 균형**이다.
모든 쿼리에 커버링 인덱스를 만들려 하면 오히려 쓰기 성능이 나빠진다.
자주 실행되고, 실행 비용이 높은 쿼리를 대상으로 선택적으로 적용한다.
