# N+1 문제

## 정의
N+1 문제는 연관 엔티티를 조회할 때, 처음 1번 조회 후 각 행마다 추가 쿼리(N번)가 발생해 총 `1 + N`번 쿼리가 실행되는 성능 문제입니다.
주로 JPA 지연 로딩(LAZY) 환경에서 목록 조회 시 자주 나타납니다.

## 특징
- 개발 초기에는 데이터가 적어 체감이 어렵고, 운영 트래픽에서 갑자기 병목이 됩니다.
- API 응답 시간이 데이터 건수에 비례해 증가합니다.
- DB CPU, 커넥션 점유 시간, 네트워크 왕복(RTT)을 동시에 증가시킵니다.

## 실제 사용 예시(사례)
예: 게시글 100건을 조회하고, 각 게시글 작성자 이름을 응답에 포함하는 경우

- 게시글 목록 조회 1번
- 각 게시글 작성자 조회 100번
- 총 101번 쿼리

해결 접근:

- Fetch Join 사용
- EntityGraph 사용
- 조회 전용 DTO 프로젝션으로 필요한 컬럼만 한 번에 조회

## 실무에서 자주 하는 실수
- `@ManyToOne(fetch = EAGER)`로 전역 해결하려는 시도
- 단순히 `hibernate.default_batch_fetch_size`만 키우고 쿼리 구조를 점검하지 않는 방식
- 컨트롤러 직렬화 시점에 지연 로딩이 터지는 문제를 방치하는 방식

## 점검 방법
- SQL 로그에서 동일 패턴 쿼리가 반복되는지 확인
- 슬로우 쿼리/애플리케이션 프로파일링으로 병목 구간 확인
- 핵심 조회 API에 대해 쿼리 횟수 단위 테스트 또는 통합 테스트 도입

## 보조 개념 정리
- Fetch Join: 연관 엔티티를 조인해 한 번에 조회하는 JPA 문법
- 지연 로딩(LAZY): 실제 접근 시점에 연관 엔티티를 조회하는 전략
- 배치 페치: 연관 엔티티를 묶어서 조회해 쿼리 수를 줄이는 전략
