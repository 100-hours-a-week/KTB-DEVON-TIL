# 키워드 문서 작성 가이드

## 목적
키워드 문서는 "면접 답변 가능 수준"에서 시작해, 필요 시 "실무 적용/성능 튜닝 수준"까지 확장하는 것을 목표로 합니다.

## 분량 기준(난이도별)

| 레벨 | 대상 | 권장 분량 | 필수 요소 |
|---|---|---|---|
| L1 기본 | DNS, HTTP, JVM 같은 기초 개념 | 짧게(핵심 위주) | 정의, 핵심 특징, 간단 사례 |
| L2 실무 | JWT, N+1, 트랜잭션 등 실무 빈출 | 중간 | 원리 + 실수 패턴 + 점검 방법 |
| L3 심화 | 인덱스, 락, GC, 쿼리 최적화 등 | 길게(구조화) | 원리, 비교표, 체크리스트, 안티패턴 |

## 마크다운 작성 규칙
- 제목은 키워드명으로 통일합니다.
- 본문은 다음 4개 섹션을 기본으로 유지합니다.
  - `## 정의`
  - `## 특징`
  - `## 실제 사용 예시(사례)`
  - `## 보조 개념 정리`
- 내용이 많아지면 아래 섹션을 추가합니다.
  - `## 동작 원리(요약)`
  - `## 종류와 선택 기준`
  - `## 성능 검증 방법`
  - `## 안티패턴`
- 비교 내용은 표(`| ... | ... |`)로 정리합니다.
- 실행/설정 예시는 코드 블록으로 작성합니다.

## 문서 품질 체크리스트
- 이 문서만 읽어도 "한 줄 정의"를 말할 수 있는가?
- 실무에서의 장단점/트레이드오프가 드러나는가?
- 오해하기 쉬운 포인트(안티패턴)를 포함했는가?
- 다음 학습 주제로 연결되는 보조 개념이 정리되어 있는가?

## 권장 운영 방식
1. 처음에는 L1으로 빠르게 작성합니다.
2. 프로젝트에서 실제로 부딪힌 주제만 L2/L3로 승격합니다.
3. 성능/장애 관련 키워드는 표와 체크리스트를 반드시 포함합니다.
