# 인덱스(Index)

## 정의
데이터베이스 인덱스는 테이블의 특정 컬럼 값을 기준으로 **검색 경로를 미리 정리한 자료구조**입니다.
책의 맨 뒤 색인처럼, 전체 데이터를 처음부터 끝까지 읽지 않고 원하는 레코드를 빠르게 찾기 위해 사용합니다.

핵심은 다음 한 줄로 정리할 수 있습니다.

> 인덱스는 "조회 성능"을 얻는 대신, "쓰기 비용"과 "저장 공간"을 추가로 지불하는 구조입니다.

## 특징
- 조회(`SELECT`) 성능을 크게 개선할 수 있습니다.
- 정렬(`ORDER BY`), 범위 조회(`BETWEEN`, `>`, `<`) 성능에도 영향을 줍니다.
- 인덱스가 많을수록 `INSERT/UPDATE/DELETE` 시 유지 비용이 증가합니다.
- 잘못 설계된 인덱스는 오히려 성능을 악화시킬 수 있습니다.

## 인덱스가 필요한 신호
아래 상황이 자주 발생하면 인덱스를 검토합니다.

- 특정 조건으로 반복 조회되는 쿼리가 많다.
- 응답 시간의 대부분이 DB 조회에 사용된다.
- `WHERE + ORDER BY + LIMIT` 형태의 API가 핵심 경로다.
- 테이블이 커지면서 Full Scan 비용이 급격히 증가한다.

## 동작 원리(요약)
대부분의 RDBMS는 B-Tree/B+Tree 기반 인덱스를 사용합니다.

1. 인덱스 루트에서 시작해 조건에 맞는 키 범위를 탐색한다.
2. 리프 노드에서 실제 행 위치(또는 PK)를 확인한다.
3. 필요한 경우 테이블 본문(클러스터/힙)을 추가 조회한다.

즉, "인덱스 탐색 + 데이터 접근"이 "테이블 전체 스캔"보다 유리할 때 성능 이점이 생깁니다.

## 인덱스 종류와 선택 기준

| 종류 | 목적 | 장점 | 주의점 |
|---|---|---|---|
| 기본 키 인덱스(PK) | 행 고유 식별 | 고유성 보장, 조인 기준 안정 | 키가 크면 보조 인덱스 비용 증가 |
| 단일 컬럼 인덱스 | 특정 컬럼 조건 조회 | 단순하고 유지 비용 예측 쉬움 | 복합 조건에서 한계 |
| 복합 인덱스 | 다중 컬럼 조건/정렬 최적화 | 핵심 쿼리 성능 개선 폭 큼 | 컬럼 순서 설계가 매우 중요 |
| 유니크 인덱스 | 중복 방지 + 조회 | 데이터 무결성 보장 | 쓰기 시 충돌 비용 존재 |
| 커버링 인덱스 | 테이블 접근 최소화 | 랜덤 I/O 감소 | 인덱스 크기 증가 |
| 부분/필터드 인덱스 | 특정 조건만 인덱싱 | 저장/유지 비용 절감 | 조건과 쿼리 일치 필요 |

## 복합 인덱스 설계 핵심
복합 인덱스는 **왼쪽 접두사(Leftmost Prefix)** 규칙을 이해해야 합니다.

예시 인덱스:

```sql
CREATE INDEX idx_orders_user_status_created
ON orders(user_id, status, created_at DESC);
```

잘 활용되는 조건 예시:

- `WHERE user_id = ?`
- `WHERE user_id = ? AND status = ?`
- `WHERE user_id = ? AND status = ? ORDER BY created_at DESC`

비효율 가능성이 큰 조건 예시:

- `WHERE status = ?` (선행 컬럼 `user_id` 조건이 없음)

설계 실무 팁:

- 등호 조건(`=`) 컬럼을 앞쪽에 배치
- 범위 조건(`>`, `<`, `BETWEEN`)은 뒤쪽에 배치
- 정렬 컬럼은 실제 쿼리 `ORDER BY`와 방향까지 맞춤

## 실제 사용 예시(사례)

### 1) 채팅 메시지 최신 목록 조회
API 패턴:

```sql
SELECT id, conversation_id, sender_id, content, created_at
FROM chat_message
WHERE conversation_id = ?
ORDER BY created_at DESC
LIMIT 20;
```

권장 인덱스:

```sql
CREATE INDEX idx_chat_message_conversation_created
ON chat_message(conversation_id, created_at DESC);
```

효과:

- 대화방별 최신 메시지 조회에서 정렬/제한 비용 감소
- 핫 경로 응답 시간(P95/P99) 안정화

### 2) 주문 이력 페이징 조회
API 패턴:

```sql
SELECT id, user_id, total_price, created_at
FROM orders
WHERE user_id = ? AND created_at < ?
ORDER BY created_at DESC
LIMIT 30;
```

권장 인덱스:

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at DESC);
```

효과:

- 커서 기반 페이지네이션과 결합 시 Offset 방식보다 안정적
- 페이지가 깊어져도 성능 하락 폭이 작음

## 성능 검증 방법
인덱스는 "만들었다"가 아니라 "검증했다"까지 해야 합니다.

1. 실행 계획(`EXPLAIN`)에서 접근 방식 확인
2. 실제 트래픽과 유사한 파라미터로 측정
3. 읽기 개선과 쓰기 비용 증가를 함께 비교

간단 체크리스트:

- `type=ALL`(Full Scan)에서 `range/ref`로 개선되었는가?
- 정렬이 filesort 없이 처리되는가?
- 목표 API의 P95/P99가 개선되었는가?
- 쓰기 TPS와 락 대기 시간이 허용 범위인가?

## 안티패턴
- "느리다"는 이유로 무작정 인덱스를 추가하는 방식
- 비슷한 목적의 중복 인덱스를 다수 유지하는 방식
- 저선택도 컬럼(예: 성별, 상태 2~3개 값) 단독 인덱스 남발
- 쿼리 패턴이 바뀌었는데 오래된 인덱스를 정리하지 않는 방식

## 보조 개념 정리
- 선택도(Selectivity): 조건이 대상을 얼마나 좁히는지 나타내는 정도
- 클러스터드 인덱스: 인덱스 순서와 실제 데이터 저장 순서가 밀접한 구조
- 커버링 인덱스: 쿼리 필요한 컬럼을 인덱스만으로 만족하는 방식
- 실행 계획(Execution Plan): DB 옵티마이저가 선택한 접근 전략
