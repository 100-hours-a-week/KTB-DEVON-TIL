# 채팅 이력 조회 캐싱 설계

## 1. 배경과 목표
채팅 이력 조회는 읽기 트래픽이 매우 높고, 데이터는 대부분 append-only 성격을 가진다. 따라서 캐싱은 단순 도입이 아니라 조회 패턴 중심으로 설계해야 한다.

핵심 목표는 다음 3가지다.

- 최신 메시지 조회(Hot Path) 응답 속도 최적화
- 과거 메시지 페이지 조회(Cold Path) 비용 절감
- 무효화 전략 단순화로 운영 안정성 확보

## 2. 데이터/조회 특성 정리
채팅 이력은 일반적으로 아래 특성을 가진다.

- Append-only 데이터
- 최신 메시지 조회가 가장 빈번
- 과거 메시지는 페이지네이션 기반 조회
- 수정/삭제는 드물거나 soft delete 중심

이 특성을 기반으로 캐시 계층을 분리한다.

- 최신 N개 캐시: 높은 조회 빈도 대응
- 과거 페이지 캐시: 반복 조회 비용 절감
- 예외 케이스(수정/삭제): 보수적 무효화

## 3. 권장 캐시 구조
Conversation 단위로 키를 분리해 관리한다.

- 최신 메시지 키: `chat:{conversationId}:latest`
- 페이지 키(권장): `chat:{conversationId}:before:{cursor}`
- 버전 적용 시: `chat:{conversationId}:v{version}:latest`

핵심 원칙은 대화방 전체를 통째로 캐싱하지 않는 것이다. 메시지 단위/페이지 단위로 쪼개야 무효화 비용이 작다.

## 4. Hot Path 설계: 최신 메시지 캐싱
### 4.1 전략
- 대화방별 최근 50~100개 메시지를 Redis List로 유지
- 새 메시지 수신 시 `LPUSH` 후 `LTRIM`으로 최대 길이 제한

### 4.2 예시 흐름
1. 클라이언트가 최근 메시지 조회 요청
2. Redis hit 시 즉시 반환
3. miss 시 DB 조회 후 Redis 세팅하여 반환

### 4.3 기대 효과
- 조회 트래픽의 대부분을 캐시가 흡수
- DB 부하 급감
- 정렬 비용 최소화(List 순서 사용)

## 5. Cold Path 설계: 페이지네이션 캐싱
### 5.1 Offset 기반(비권장)
Offset 기반 페이지 캐시는 채팅에 부적합하다.

- 새 메시지 추가 시 페이지 경계가 밀림
- 기존 캐시 무효화 범위가 커짐

### 5.2 Cursor 기반(권장)
Cursor를 `messageId` 또는 `createdAt(+messageId)`로 구성한다.

- 키 예시: `chat:{conversationId}:before:{cursor}`
- append-only 특성에서 페이지 안정성이 높음
- 기존 페이지 캐시를 오래 재사용 가능

## 6. 무효화 전략
### 6.1 일반 케이스(메시지 추가)
- 최신 캐시만 갱신
- 과거 페이지 캐시는 유지

### 6.2 예외 케이스(수정/삭제)
수정/삭제가 발생하면 보수적으로 처리한다.

- 단순 전략: conversation 단위 캐시 무효화
- 운영 전략: versioning으로 자연 교체

## 7. Versioning 패턴
수정/삭제 발생 시 key version을 증가시켜 신규 캐시 공간을 사용한다.

- 키 예시: `chat:{conversationId}:v{version}:before:{cursor}`
- 기존 버전 캐시는 TTL 만료로 자연 소거
- 대량 `DEL` 의존도를 줄여 운영 리스크 완화

## 8. TTL/메모리 정책
추천 TTL 범위는 다음과 같다.

- 최신 캐시(`latest`): 1~5분
- 과거 페이지 캐시: 5~30분
- 비활성 대화방: 1~5분

메모리 정책은 Redis eviction(LRU/LFU)와 함께 설계하되, TTL 없는 무한 캐시는 금지한다.

## 9. 대규모 트래픽 확장 포인트
### 9.1 Write Through vs Write Behind
- Write Through: 일관성 우선, 지연 증가 가능
- Write Behind: 성능 우선, 유실/재처리 설계 필요

서비스 특성에 따라 동기 저장 또는 이벤트 기반 비동기 저장을 선택한다.

### 9.2 이벤트 연계
메시지 생성 이벤트를 아래와 연계해 fan-out/후처리를 분리한다.

- Redis Pub/Sub
- Redis Streams
- Kafka

## 10. 안티패턴 (심각한 성능 저하 유발)
아래 방식은 실제 운영에서 지연 급증, Redis CPU 스파이크, 타임아웃을 만들기 쉬운 패턴이다.

### 10.1 `KEYS chat:{conversationId}:*`를 요청 경로에서 사용하는 설계
- 문제: `KEYS`는 전수 스캔 성격이라 데이터가 커질수록 비용이 급증한다.
- 영향: 하나의 API 요청이 Redis 단일 스레드를 오래 점유해 전체 지연을 유발한다.
- 대안: 버전 키(`v{version}`)로 무효화하거나, 사전에 관리하는 정확한 키 목록만 삭제한다.

### 10.2 대화방 전체를 단일 JSON 문자열로 저장/갱신하는 설계
- 문제: 메시지 1건 추가에도 직렬화/역직렬화 + 전체 문자열 재저장이 발생한다.
- 영향: payload가 커질수록 네트워크/CPU/메모리 복사 비용이 `O(n)`으로 증가한다.
- 대안: 최신 N개는 List, 과거는 cursor 페이지 키로 분할 저장한다.

### 10.3 매 요청마다 `LRANGE 0 -1` 또는 `ZRANGE 0 -1`로 전체 이력을 읽는 설계
- 문제: 전체 조회는 메시지 수에 비례해 커지는 `O(n)` 접근이다.
- 영향: 방 하나가 커질수록 응답 시간이 선형 증가하고 hot room에서 병목이 발생한다.
- 대안: 최신 조회는 고정 길이(`LRANGE 0 99`), 과거는 cursor 단위 페이지 크기(예: 20~50)로 제한한다.

### 10.4 거대한 Set/Hash에 대해 `SMEMBERS`/`HGETALL`을 반복 호출하는 설계
- 문제: 전체 필드/멤버를 매번 가져오는 구조는 전형적인 `O(n)` 연산이다.
- 영향: 메모리 사용량과 응답 직렬화 비용이 동시에 커지고 P99 지연이 급격히 악화된다.
- 대안: 필요한 필드만 조회(`HMGET`)하고, 대화 단위 키 분리로 키 크기 자체를 작게 유지한다.

### 10.5 페이지 이동마다 `SORT`를 실행해 정렬을 Redis에 위임하는 설계
- 문제: 대량 데이터 정렬은 `O(n log n)`으로 비싸며, 요청 경로에서 반복 시 치명적이다.
- 영향: 트래픽 피크 시 Redis CPU가 정렬에 잠식되어 캐시 서버가 병목이 된다.
- 대안: 쓰기 시점에 정렬 상태를 유지(List/ZSet score)하고 조회 시 재정렬을 없앤다.

### 10.6 대량 키 삭제를 동기 `DEL`로 한 번에 수행하는 설계
- 문제: 큰 키(또는 많은 키) 동기 삭제는 이벤트 루프 블로킹처럼 지연을 만든다.
- 영향: 무효화 순간 전체 트래픽 지연이 튀고 timeout이 발생할 수 있다.
- 대안: 버전 증가 방식으로 무효화하거나 점진적 삭제 전략을 사용한다.

### 10.7 TTL 없이 영구 캐시를 유지하는 설계
- 문제: cold key가 누적되어 메모리 단편화/eviction 압력이 커진다.
- 영향: eviction 폭증, hit ratio 하락, 예측 불가능한 성능 저하로 이어진다.
- 대안: 데이터 온도에 맞는 TTL(최신 짧게, 과거는 중간)과 eviction 정책을 함께 운영한다.

### 10.8 Offset 기반 페이지 캐시를 핵심 전략으로 쓰는 설계
- 문제: 신규 메시지 추가 시 페이지 경계가 밀려 캐시 재사용률이 급감한다.
- 영향: 실질적으로 cache miss가 늘고 DB fallback이 급증한다.
- 대안: cursor 기반(`before:{cursor}`)으로 페이지 안정성을 확보한다.

## 11. 기본 권장안(현실적인 시작점)
대부분 서비스에서 바로 적용 가능한 균형안은 다음과 같다.

- Redis List로 최신 100개 유지
- 과거 조회는 Cursor 기반 페이지 캐시
- TTL 5~15분부터 시작해 트래픽에 맞게 튜닝
- 수정/삭제는 드물다는 가정 하에 예외 시 conversation invalidate 또는 version 증가

## 12. 구현 전 확인 체크리스트
- 평균 대화 길이와 방당 메시지 수
- 읽기/쓰기 비율
- 다중 서버 환경 여부
- Redis 단일/클러스터 구성 여부
- 메시지 수정/삭제 정책(허용 범위)
- 영구 저장 동기/비동기 전략과 재처리 정책

위 항목의 실제 수치를 확보하면, 소규모/대규모/DDD 계층화 관점으로 설계를 확장할 수 있다.

## 13. 포트폴리오 시나리오(통합 버전)
아래는 본 문서를 포트폴리오 소재로 전개할 때 사용할 통합 시나리오다. 핵심은 단일 개선 포인트가 아니라, 실제 장애 흐름처럼 원인 분리와 단계별 개선을 보여주는 것이다.

### 서비스 특성 반영
- AI 채팅 서비스이며 최근 대화 이력 20개 조회가 가장 빈번하다.
- 과거 이력 조회는 무한 스크롤 기반이며 우선 최근 100개를 캐싱 대상으로 둔다.
- 신규 메시지가 지속적으로 추가되는 append-heavy 패턴이다.

### 1단계: O(n)/Offset/DEL 기반 초기 설계
- 최신/과거 조회를 분리하지 않고 단일 캐시 경로로 처리
- offset 기반 페이지 캐시 사용
- 수정/삭제 시 광범위 `DEL` 무효화
- 일부 요청에서 전체 조회성 접근(`LRANGE 0 -1` 등) 사용

### 2단계: 피크 트래픽에서 장애 재현(`k6`)
- 시나리오 A: 최근 20개 조회 집중(핫 경로)
- 시나리오 B: 무한 스크롤 조회 + 신규 메시지 동시 유입
- 시나리오 C: TTL 동시 만료 구간에서 burst 트래픽 인가

### 3단계: 원인 분리
- 데이터 구조: 전체 조회성 명령으로 인한 `O(n)` 비용 증가
- RTT: 요청당 Redis 왕복 횟수 과다
- 무효화: `DEL` 집중 시점의 지연 스파이크
- 직렬화: 과대 payload 직렬화/역직렬화 비용

### 4단계: 순차 개선
- 커서 전환: offset -> cursor(`before:{cursor}`)로 페이지 안정성 확보
- 버저닝: `v{version}` 키 전략으로 무효화 비용 절감
- stampede 방지: TTL jitter + single-flight + stale-while-revalidate
- 직렬화 최적화: 최신 20개는 경량 DTO 캐시, 과거 페이지는 필요 필드만 제공

### 5단계: 전/후 비교
아래 지표를 동일 부하 조건에서 전/후로 비교한다.

- API 지연: `P95`, `P99`
- 캐시 효율: latest(20개) hit ratio, history(무한스크롤) hit ratio
- 저장소 부하: DB/DynamoDB reads per request, burst 시 read spike
- Redis 상태: CPU, ops/sec, keyspace hit/miss, slowlog
- 비용 관점: 요청당 read cost, 피크 구간 비용 상승률

### 최종 구조(개선안 요약)
- `chat:{conversationId}:latest`에 최근 20개 고빈도 조회 최적화
- `chat:{conversationId}:before:{cursor}` 기반 무한 스크롤 페이지 캐시(최대 100개 범위)
- 신규 메시지 시 `LPUSH + LTRIM` 중심 갱신, 과거 페이지는 유지
- 수정/삭제 예외 시 버저닝 전환으로 안전 무효화
