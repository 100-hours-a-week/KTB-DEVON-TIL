# _3 직렬화/역직렬화 비용 병목

## 문제 정의
Redis 응답이 빨라도 payload가 크고 파싱 비용이 높으면 애플리케이션 CPU가 병목이 된다.

## 재현 조건
- latest/history 응답에 불필요 필드 다수 포함
- JSON 기반 직렬화/역직렬화 반복
- 고부하 시점에서 앱 CPU와 GC 압력 관찰

## 개선 전략
- MessagePack/ProtoBuf 검토
- hot path는 경량 DTO(필수 필드만) 캐시
- 상세 필드는 lazy fetch로 분리

## 검증 지표
- 앱 CPU 사용률
- 응답 바이트 크기
- `P95/P99`, GC 지표

## 포트폴리오 포인트
"Redis는 빨랐는데 앱이 느렸다"는 반직관 문제를 데이터 포맷 최적화로 해결한 사례를 제시할 수 있다.
