# 통합 5단계 시나리오

## 1단계: O(n)/Offset/DEL 기반 초기 설계
- 최신/과거 조회를 분리하지 않음
- offset 기반 페이지 캐시
- 수정/삭제 시 광범위 `DEL`
- 일부 요청에서 전체 범위 조회(`LRANGE 0 -1`)

## 2단계: 피크 트래픽 장애 재현 (`k6`)
- A: 최신 20개 집중 조회
- B: 무한 스크롤 + 신규 메시지 동시 유입
- C: TTL 동시 만료 burst

실험 규칙:
- 동일 데이터셋, 동일 VU/Duration, 동일 warm-up
- 최소 3회 반복 후 중앙값 비교

## 3단계: 원인 분리
- 데이터 구조: `O(n)` 조회 명령 사용
- RTT: 요청당 Redis 왕복 과다
- 무효화: `DEL` 시점 지연 급등
- 직렬화: payload 과대로 앱 CPU 증가

## 4단계: 순차 개선
1. cursor 전환(`before:{cursor}`)
2. versioned key(`v{version}`)
3. stampede 방지(TTL jitter + single-flight + SWR)
4. 직렬화 경량화(최신 20개 DTO 분리)

## 5단계: 전/후 비교
- API: `P95/P99`, 에러율
- 캐시: latest/history hit ratio
- 저장소: `reads/request`, burst read spike
- Redis: CPU, `ops/sec`, slowlog
- 비용: `10k` 요청당 read cost

## 최종 설계 요약
- 최신 20개: `chat:{conversationId}:latest`
- 이력 스크롤: `chat:{conversationId}:before:{cursor}`
- 캐시 범위: 이력 100개 (페이지 단위)
- 쓰기 반영: `LPUSH + LTRIM`
- 수정/삭제: version up + lazy expiry
